## B — Bugs / Crashes

### B1 · Nil pointer crash in `tbarLegendSizeChange`

| | |
|---|---|
| **Where** | `FormMain.pas:94` |
| **Code** | `TFrmSkiaChartModel(FFrmChart).LegendSize := tbarLegendSize.Value / 100;` |
| **Why** | `FFrmChart` is `nil` until the user clicks "Re-Draw" at least once. Moving the legend-size track bar before that moment hard-casts a nil pointer and raises an Access Violation. No `Assigned(FFrmChart)` guard exists before the cast. |
| **How** | Add a guard: `if not Assigned(FFrmChart) then Exit;` at the top of `tbarLegendSizeChange`, or initialize `FFrmChart` with a default chart type in `FormCreate`. |

### B2 · Uninitialized `LTotal` in Pie `skChartMouseDown` else-branch

| | |
|---|---|
| **Where** | `SkiaChart.View.Pie.pas:184–189` |
| **Code** | `else begin FSelectedItem := -1; UpdateLegend(FSelectedItem, LCenter, LRadius, LTotal); end;` |
| **Why** | `LTotal` is only computed inside the `if LDistance <= LRadius then` branch (lines 159–164). In the `else` branch (click outside the pie), `LTotal` is a declared but uninitialized local `Single` — its value is whatever happens to be on the stack. `UpdateLegend` receives garbage. It exits early because `AIndex = -1`, so it does not crash today, but this is formally undefined behaviour and fragile if `UpdateLegend`'s guard is ever changed. |
| **How** | Initialize `LTotal := 0;` at the top of the procedure (alongside `LAngle`, `LDistance`, etc.), before the conditional branching. |

### B3 · Color-index wrapping handles only one cycle

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas:160–163` |
| **Code** | `if LIndex > Pred(Length(CAryColors)) then LIndex := LIndex - Length(CAryColors);` |
| **Why** | The palette has 11 colors. When 12+ items are added the index is wrapped by subtracting 11 — correct for the first overflow. But when 23+ items are added (second overflow), the subtraction produces an index that is still > 10, causing an out-of-bounds read on `CAryColors`. |
| **How** | Replace the subtraction with a modulo: `LIndex := LIndex mod Length(CAryColors);` — handles any number of items correctly and is self-documenting. |

### B4 · Pie animation speed accumulates across restarts

| | |
|---|---|
| **Where** | `SkiaChart.View.Pie.pas:200–209` |
| **Code** | `FAnimationSpeed := FAnimationSpeed * 1.05;` executes unconditionally after the `>= 360` cap check. |
| **Why** | When the animation finishes, the cap is applied (`FAnimationProgress := 360`) but the speed multiplication runs one more time, leaving `FAnimationSpeed` at its peak accelerated value. `StartAnimation` does reset it back to 5, so today this is benign. However, any code path that calls `tmrAnimation.Enabled := True` without going through `StartAnimation` would start the next animation at the stale (high) speed, causing it to jump to completion almost instantly. |
| **How** | Correct shape — swap `Exit` for an `else` so the speed multiplication is the thing that gets skipped, and the `Redraw` still fires: |
```delphi
procedure TFrmSkiaChartPie.tmrAnimationTimer(Sender: TObject);
begin
  FAnimationProgress := FAnimationProgress + FAnimationSpeed;
  if FAnimationProgress >= 360 then
  begin
    tmrAnimation.Enabled := False;
    FAnimationProgress := 360;
  end
  else
    FAnimationSpeed := FAnimationSpeed * CAccelerationRate;
  skChart.Redraw; // always runs, final frame draws correctly
end;
```
Semantics now:
- **Mid-animation tick:** accumulate, accelerate, redraw
- **Final tick:** accumulate, cap at 360, disable timer, **do not** accelerate (keeps the field clean for next run), redraw (draws the complete circle)

### B5 · Bar hit-test coordinate system differs from draw coordinate system

| | |
|---|---|
| **Where** | `SkiaChart.View.Bars.pas:147–150` (mouse handler) vs. `DoChartDraw:75–78` (draw) |
| **Why** | `DoChartDraw` computes bar positions relative to `ADest` (the canvas destination rect passed by `TSkPaintBox`). `skChartMouseDown` computes the same positions relative to `skChart.Width` / `skChart.Height` (the control's logical size). These are identical only when `ADest` has no viewport offset. If `TSkPaintBox` ever introduces padding or a scaled `ADest`, the calculated bar rects in mouse handling will be offset from the drawn rects, causing click misses. |
| **How** | Store the computed bar rects as a field (`FLastBarRects: TArray<TRectF>`) populated during `DoChartDraw` and read directly in `skChartMouseDown` — eliminating the duplication and ensuring both code paths use the same coordinates. (Also resolves Q1.) |

## M — Memory / Object Lifetime

### M1 · `TLayoutLegend` dual-ownership: component owner AND FMX parent

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas:143–149` |
| **Code** | `var LLyt := TLayoutLegend.Create(Self);` then `LLyt.Parent := lytLegend;` then `FObjLstLegend.Add(LLyt);` |
| **Why** | In FMX, passing `Self` as owner adds `LLyt` to `Self.FOwnedComponents` — freed when `Self` is destroyed. Setting `Parent := lytLegend` also registers `LLyt` as a child in `lytLegend.Children` — also freed when `lytLegend` is destroyed. When `TFrmSkiaChartModel` teardown runs, both ownership chains converge on the same `TLayoutLegend` instances. Depending on teardown order this can produce a double-free (AV) or a silent use-after-free. `ReportMemoryLeaksOnShutdown := True` in the project already hints the developers are watching for this. |
| **How** | Pass `nil` as owner (`TLayoutLegend.Create(nil)`) and rely solely on the FMX parent chain for lifetime management. Remove the manual `ExtractAt` loop from the destructor — it becomes unnecessary once ownership is unambiguous. Alternatively, keep `Self` as owner and never set a `Parent` (manually position and draw legend items), but that is far more invasive. |

---

### M2 · Destructor calls `ExtractAt(0)` but discards the extracted object

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas:117–118` |
| **Code** | `while FObjLstLegend.Count > 0 do FObjLstLegend.ExtractAt(0);` |
| **Why** | `TObjectList.ExtractAt` removes the item from the list and returns it — it does NOT free the item. The return value is silently discarded. The items are not leaked only because the FMX parent chain (`lytLegend`) frees them. If `lytLegend` is freed before this destructor code runs (or if the parent relationship is removed), every extracted `TLayoutLegend` leaks. This is a latent memory leak hiding behind a currently-lucky destruction order. |
| **How** | Once M1 is resolved (use `nil` owner), this loop can be removed entirely — the FMX parent handles cleanup. If dual-ownership is kept, use `FObjLstLegend.Clear` (which calls `Free` on each item when `OwnsObjects = True`) instead of the extract loop, but guard against the parent double-free first. |

---

### M3 · Empty `except` in destructor silently swallows all errors

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas:116–121` |
| **Code** | `try ... FreeAndNil(FObjLstLegend); except end;` |
| **Why** | An unconditional `except` with an empty handler is one of the most dangerous patterns in Delphi. Any exception during cleanup — including a double-free AV — is silently discarded. The destructor then calls `inherited` as if nothing happened, potentially leaving the object in a half-destroyed state. With `ReportMemoryLeaksOnShutdown := True` already enabled in the project, this pattern directly defeats the project's own leak-detection intent. |
| **How** | Remove the try/except entirely. After resolving M1 and M2, the cleanup body becomes trivial (`FreeAndNil(FObjLstLegend)`) and does not need exception protection. If some protection is genuinely needed, at minimum log or re-raise — never silently swallow in a destructor. |

### M4 · Empty `except` in `DoChartDraw` hides `UpdateLegend` errors

| | |
|---|---|
| **Where** | `SkiaChart.View.Pie.pas:122–126` |
| **Code** | `try UpdateLegend(FSelectedItem, LCenter, LRadius, LTotal); except end;` |
| **Why** | `UpdateLegend` accesses `FItems[AIndex]`, computes float divisions, and sets FMX control properties. Any exception (nil-dereference, division by zero with a future `LTotal = 0` path, FMX control state error) is swallowed. Because `DoChartDraw` is called from the Skia paint loop, a real crash would be suppressed every frame while the chart silently renders incorrectly. This makes debugging extremely difficult. |
| **How** | Remove the try/except. Fix the underlying risks in `UpdateLegend` directly (guard `ATotal = 0`, guard `AIndex` bounds). The Skia `OnDraw` callback does not require exception suppression. |

### M5 · `Clear` does not reset `FSelectedItem`

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas:91–102` |
| **Why** | After `Clear` + re-add of items (without calling `StartAnimation`), `FSelectedItem` retains the last selection index. If the new dataset has fewer items, the stale index points past `High(FItems)` — an out-of-bounds read in `UpdateLegend`'s bounds check only protects one of the code paths (the check is `AIndex >= Length(FItems)` which guards the popup, but `FSelectedItem` being non-negative still affects `DoChartDraw`'s pass through `UpdateLegend`). |
| **How** | Add `FSelectedItem := -1;` inside `Clear`, alongside `SetLength(FItems, 0)` and `FObjLstLegend.Clear`. |

## P — Performance

### P1 · `TLayoutLegend.Painting` mutates `Width` and child sizes every paint cycle

| | |
|---|---|
| **Where** | `SkiaChart.Controller.Chart.pas:167–174` |
| **Code** | `TControl(FRectColor.Parent).Width := FRectColorWidth * FLegendSize;` `FRectColor.Height := FRectColorHeight * FLegendSize;` `FLbl.TextSettings.Font.Size := 12 * FLegendSize;` `Width := FLbl.Position.X + FLbl.Width + 10;` |
| **Why** | Assigning to `Width`, `Height`, or `Font.Size` inside `Painting` marks the parent layout dirty and schedules another paint. FMX's paint guard prevents infinite recursion, but it still triggers unnecessary invalidation passes every frame when `FLegendSize` has not changed. On legend-heavy charts this multiplies into many redundant layout calculations per second. |
| **How** | **Split by concern, do not remove `Painting` entirely.** The original override was doing two logically-distinct jobs: (A) scale child sizes by `FLegendSize` — change-driven, belongs in `SetLegendSize` with a `SameValue` guard; (B) set `Width := FLbl.Position.X + FLbl.Width + 10` — paint-driven, because `FLbl.AutoSize = True` means `FLbl.Width` is resolved lazily during the FMX paint pass. Moving (B) into the setter reads a **stale** `FLbl.Width` and the container never grows to fit the new font, which breaks the flow-layout in `lytLegendPainting`. Correct shape: keep `Painting` but make the `Width` write idempotent (`if not SameValue(Width, LNewWidth) then Width := LNewWidth;`) to break the cascading-repaint loop while still reacting to each new `FLbl.Width` FMX produces. |

### P2 · `lytLegend.Height` assigned inside the per-item layout loop

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas:178` |
| **Code** | `lytLegend.Height := LHeightBase;` (inside `for i := 0 to Pred(FObjLstLegend.Count)`) |
| **Why** | Every `Height` assignment on a FMX control calls `Repaint` on the parent chain. This line runs once per legend item on every paint cycle (because it is inside `lytLegendPainting`). For 7 items, that is 7 redundant repaints per frame before the final height is even known. |
| **How** | Compute the final `LHeight` value through the entire loop first (without touching `lytLegend.Height`), then assign once after the loop: `lytLegend.Height := LHeight + 5;`. |

### P3 · `TSkPathBuilder` + `ISkPath` allocated per bar per draw frame

| | |
|---|---|
| **Where** | `SkiaChart.View.Bars.pas:104–111` |
| **Code** | `LPathBuilder := TSkPathBuilder.Create; try LPathBuilder.AddRect(LRect); LPath := LPathBuilder.Detach; ACanvas.DrawPath(LPath, LPaint); finally LPathBuilder := nil; end;` |
| **Why** | Drawing a rectangle does not require a path object. `ISkCanvas` provides `DrawRect(const ARect: TRectF; const APaint: ISkPaint)` directly. Each bar currently allocates two interface-counted Skia objects per frame, where zero allocations are needed. For 7 bars at 60 fps this is 840 unnecessary allocations per second. |
| **How** | Replace the `TSkPathBuilder` block with `ACanvas.DrawRect(LRect, LPaint)`. If rounded corners are ever needed, use `ACanvas.DrawRoundRect`. |

### P4 · Pie animation timer fires at 3 ms (~333 Hz) for a 60 Hz display

| | |
|---|---|
| **Where** | `SkiaChart.View.Pie.pas:194–195` |
| **Code** | `FAnimationSpeed := 5; tmrAnimation.Interval := 3;` |
| **Why** | Windows `TTimer` at 3 ms fires at approximately 333 events per second. Each event calls `skChart.Redraw`, which schedules a full Skia repaint. A 60 Hz display can only show 60 frames per second; the other 273 redraws are discarded by the compositor but still consume CPU. The acceleration factor (`* 1.05`) means the pie reaches 360° in roughly 70 ticks regardless of interval — dropping to 16 ms (60 Hz) halves CPU cost with no visible quality difference. |
| **How** | Change `tmrAnimation.Interval := 3` to `tmrAnimation.Interval := 16` and adjust `FAnimationSpeed` or the acceleration factor proportionally so total animation duration remains the same. Alternatively, use `TSkAnimatedPaintBox` which hooks into the vsync signal directly. |

## Q — Code Quality / Correctness

### Q1 · Bar dimension calculation duplicated in three places

| | |
|---|---|
| **Where** | `SkiaChart.View.Bars.pas:75–113` (`DoChartDraw`) · `146–169` (`skChartMouseDown`) · `250–270` (`UpdateLegend`) |
| **Why** | The identical formula — `LBarWidth := (Width - 2 * CBarMargin - (LEnabledCount - 1) * CBarSpacing) / LEnabledCount`, `LTotalWidth`, `LStartX`, `LMaxHeight` — is copy-pasted across all three methods. A change to the formula (e.g., a new margin constant) must be applied three times. A bug in one location (e.g., B5) creates invisible divergence across the others. This is already causing the coordinate mismatch described in B5. |
| **How** | Extract a private `CalcBarRects(const AWidth, AHeight: Single): TArray<TRectF>` that accepts the relevant dimensions and returns the full array of bar rects. All three call sites consume this single source of truth. Cache the last result in `FLastBarRects` (see B5). |

### Q6 · `inherited` called in non-virtual event handlers

| | |
|---|---|
| **Where** | `SkiaChart.View.Bars.pas:131` (`skChartMouseDown`), `201` (`tmrAnimationTimer`) · `SkiaChart.View.Pie.pas:140` (`skChartMouseDown`), `199` (`tmrAnimationTimer`) |
| **Why** | These are plain procedures assigned to FMX event properties — they are not declared `virtual` or `override` in the base class `TFrmSkiaChartModel`. Calling `inherited` in this context resolves to `TFrame`'s empty implementation (effectively a no-op). If a maintainer later adds a same-named virtual method to `TFrmSkiaChartModel` and forgets these `inherited` calls exist, unexpected double-execution will occur. The calls give a false impression of polymorphism. |
| **How** | Remove all four `inherited;` calls from these event handlers. If shared base behaviour is actually needed, declare a protected virtual method (e.g., `procedure DoAnimationTick; virtual;`) and call it explicitly. |

### Q7 · `OnLegendTap` uses `LEnabledCount := 2` as a sentinel value

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas:203` |
| **Code** | `var LEnabledCount := 2; if FItems[TLayoutLegend(Sender).&Index].Enabled then begin LEnabledCount := 0; for ...` |

## A — Architecture / Extensibility

### A1 · Selected-item popup not repositioned on chart resize

| | |
|---|---|
| **Where** | `SkiaChart.View.Bars.pas` (`UpdateLegend:287–309`) · `SkiaChart.View.Pie.pas` (`UpdateLegend:248–274`) |
| **Why** | The popup (`lytSelectedItem`) is placed at an absolute `Position.X/Y` calculated from bar or slice geometry at the moment of selection. If the user resizes the window while the popup is visible, the popup floats at its original coordinates — misaligned from the bar or slice it annotates. FMX's layout system does not automatically reposition absolute-positioned controls. |
| **How** | Override `DoResized` (or handle `OnResize` on `skChart`) and call `UpdateLegend(FSelectedItem)` to recalculate and reposition the popup whenever the control's size changes. Guard with `if FSelectedItem >= 0 then`. |

### A2 · `TFrmSkiaChartModel` conflates data model with view/layout

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas` — the entire unit |
| **Why** | `TFrmSkiaChartModel` is simultaneously: the data store (`FItems: TArray<TChartItem>`), the legend layout engine (`lytLegendPainting`, `FObjLstLegend`), the animation controller (`tmrAnimation`, `FAnimationProgress`), and the Skia draw dispatcher (`skChartDraw`). This violates single-responsibility. In practice it means: data logic cannot be tested without instantiating a form; adding a new chart type requires forking `TFrmSkiaChartModel`; changing the legend layout affects the data API. |
| **How** | Separate into at minimum two units: a plain `TChartDataSet` class (items, add/remove, enabled state, color palette) with no FMX dependency, and the frame class that consumes it. This is a larger refactor; the immediate benefit is that `TChartDataSet` becomes unit-testable without a GUI. |

### A3 · `TLayoutLegend` visual structure is entirely hardcoded in constructor

| | |
|---|---|
| **Where** | `SkiaChart.Controller.Chart.pas:104–137` |
| **Why** | Font size, color rect dimensions, margins, and padding are all wired into `Create`. A consumer cannot change the legend style (e.g., show icon instead of colored rectangle, different font, right-aligned text) without subclassing and overriding the constructor — which requires knowing the private child component structure. |
| **How** | At minimum, surface the current magic numbers as public properties (see Q3). A more complete approach exposes a `TLegendItemStyle` record and applies it in a `ApplyStyle(const AStyle: TLegendItemStyle)` method called after construction. |

### A4 · Animation uses manual `TTimer` + progress float instead of FMX animation framework

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas` (timer declarations) · `Bars.pas` (`tmrAnimationTimer`, `StartAnimation`) · `Pie.pas` (`tmrAnimationTimer`, `StartAnimation`) |
| **Why** | FMX ships `TFloatAnimation` and `TSkAnimatedPaintBox` precisely for this use case. The current implementation re-creates their functionality manually: a `TTimer`, a `Single` progress field, a speed constant, and a `Redraw` call per tick. The manual approach does not respect vsync, fires extra redraws (see P4), does not integrate with FMX's animation queue (pause/resume, animation groups), and requires duplicated boilerplate in every chart type. |
| **How** | Use `TSkAnimatedPaintBox` (from `FMX.Skia`) in place of `TSkPaintBox` + `TTimer`. It provides a built-in `OnAnimate` event driven by FMX's vsync-aligned animation loop, removing the need for `tmrAnimation` and `FAnimationProgress` entirely. The `AProgress` parameter in `OnAnimate` replaces `FAnimationProgress`. |

### A5 · No public event when the user selects or deselects an item

| | |
|---|---|
| **Where** | `SkiaChart.View.Model.pas` — public interface |
| **Why** | `FSelectedItem` is a private field. A consumer of the chart frame (e.g., `FormMain`) has no way to react when the user clicks a bar or slice. This forces tight coupling: any "selection changed" logic must be added inside the chart frame, breaking the encapsulation the Controller/View separation is trying to establish. |
| **How** | Add a protected virtual method `procedure DoItemSelected(AIndex: Integer); virtual;` called whenever `FSelectedItem` changes, and expose a public `OnItemSelected: TProc<Integer>` property (or typed event) so callers can attach their own handlers. |

## Extra info

### Q7 — Can `TLayoutLegend(Sender).&Index` be invalid?

In the current code, no — because:
1. `&Index` is assigned once in `ItemAdd` (Model.pas:157) and never modified afterwards.
2. The only code path that removes legends is `Clear`, which calls `FObjLstLegend.Clear` → with `OwnsObjects = True` this frees every legend **instance**, so no legend survives to fire a stale tap.
3. `FItems` is only appended to (same `ItemAdd`) or cleared (`Clear`) — never shrunk individually.

So today every `&Index` matches a valid slot in `FItems`.

**The fragility** appears the moment anyone adds a `RemoveItem(AIndex: Integer)` method that deletes from the middle of `FItems`. The surviving legends at indices `> AIndex` would have stale `&Index` values pointing past the shifted array, and `FItems[LIndex]` at the top of `OnLegendTap` would go out of bounds.

If you ever add per-item removal, the pattern to use is: after removing item K, loop over `FObjLstLegend` and decrement every `&Index` that was `> K`. Or reference items by GUID/Text instead of integer index. Not a bug today, just worth writing on a post-it for later.

### P4 — What does "drop 1.05 to 1.02 once base speed is higher" mean?

Unpack the pie animation math. Each timer tick:
```
speed := speed * CAccelerationRate   // 1.05
progress := progress + speed         // degrees
```

With `speed₀ = 5` and rate `1.05`, after `n` ticks speed is `5 * 1.05ⁿ` — doubles every ~14 ticks. Total progress traversed is roughly `5 * (1.05ⁿ⁺¹ − 1) / 0.05`. Reaches 360° around tick 30.

Interval changed from 3 ms → 16 ms (5.3× slower clock). Speed kept at 5. Result: same 30 ticks, but each tick is now ~16 ms → total animation time went from ~100 ms to ~500 ms. That may feel sluggish compared to the original.

**Two knobs to get the original feel back:**
- **Option A (keep the feel simple):** leave everything as-is — you may like the slower animation.
- **Option B (match original duration):** raise `CInitialSpeed` to ~27 (= 5 × 5.3) so total ticks stay around 30 and duration stays around 500 ms. But now each tick starts at 27° and the exponential `* 1.05` kicks in from a much bigger base — by tick 10 speed is already ~44°/tick, making the last third of the animation blur past. That's where *"drop 1.05 to 1.02"* comes in: a gentler acceleration factor keeps the "slow start, brisk finish" feel when the starting speed is already high.

Rule of thumb: **speed × acceleration^ticks should roughly cover 360°**. If you bump speed, drop acceleration; if you lower speed, raise acceleration. Current (5 / 1.05) is internally consistent — leave it alone unless the new duration bugs you.

### P1 — Why the setter-only fix broke resizing, and the correct hybrid

**The regression:** after removing `Painting` and moving every assignment into `SetLegendSize`, `TLayoutLegend` items no longer resize proportionally when `lytLegend.Width` changes. The font grows but the container's `Width` does not follow, so `lytLegendPainting` lays legends out against stale widths.

**Root cause:** the original `Painting` was doing two things that look similar but have different timing requirements.

| Line | Job | Driven by | Correct home |
|------|-----|-----------|--------------|
| `TControl(FRectColor.Parent).Width := FRectColorWidth * FLegendSize;` | Scale color-rect container | `FLegendSize` change | `SetLegendSize` (guarded) |
| `FRectColor.Height := FRectColorHeight * FLegendSize;` | Scale color rect | `FLegendSize` change | `SetLegendSize` (guarded) |
| `FLbl.TextSettings.Font.Size := 12 * FLegendSize;` | Scale label font | `FLegendSize` change | `SetLegendSize` (guarded) |
| `Width := FLbl.Position.X + FLbl.Width + 10;` | Make container wrap the label | `FLbl.Width` change (paint-resolved) | `Painting` with compare-before-assign |

The last line is the trap. `FLbl.AutoSize = True` means FMX computes `FLbl.Width` lazily, **during the paint pass**, after the new `Font.Size` flows through the text-measurement code. At the moment `SetLegendSize` runs, `FLbl.Width` still holds its previous value — so the container `Width` freezes at the pre-resize number.

**Corrected design:**

```delphi
procedure TLayoutLegend.SetLegendSize(const Value: Single);
begin
  if SameValue(FLegendSize, Value) then
    Exit;
  FLegendSize := Value;
  TControl(FRectColor.Parent).Width := FRectColorWidth * Value;
  FRectColor.Height := FRectColorHeight * Value;
  FLbl.TextSettings.Font.Size := 12 * Value;
  // Intentionally NOT setting Width here — FLbl.Width is stale until next paint.
end;

procedure TLayoutLegend.Painting;
begin
  inherited;
  var LNewWidth := FLbl.Position.X + FLbl.Width + 10;
  if not SameValue(Width, LNewWidth) then
    Width := LNewWidth;
end;
```

**Why this preserves the P1 win:**
- Job A mutations happen **once per actual `FLegendSize` change**, not per frame → no cascading repaint from font/rect writes.
- Job B still runs every paint, but `Width` is only *written* when the computed value differs from the current one → the first paint after a size change writes once (legitimate), subsequent paints no-op. No cascade.

**Why a pure-setter approach can't work:** forcing FMX to synchronously recompute `FLbl.Width` from inside `SetLegendSize` requires either a recursive paint (worse than the original) or calling private measurement APIs. FMX's AutoSize contract is "width resolves during paint" — respect it rather than fight it.
